# 備份路徑與角色更新修復說明

## 修復內容總結

本次修復解決了兩個問題：
1. ✅ 備份與還原頁面的備份目錄路徑硬編碼問題
2. ✅ 角色管理編輯更新失敗問題

## 問題 1：備份目錄路徑顯示

### 問題描述
在備份與還原頁面的注意事項中，備份目錄路徑顯示為硬編碼的 `/opt/invoice-bonus-backups`，但實際的備份目錄可能是 `/opt/project-system-backups` 或其他自訂路徑。

### 解決方案

#### 修改檔案（3 個）

1. **`src/services/BackupRestoreService.js`**
   - 新增 `getBackupDir()` 靜態方法
   - 新增 `getInstallDir()` 靜態方法
   - 供路由使用，獲取當前系統的實際備份目錄

2. **`src/routes/backupRestore.js`**
   - 調用 `BackupRestoreService.getBackupDir()` 獲取實際路徑
   - 將 `backupDir` 傳遞給視圖

3. **`src/views/backup-restore/index.ejs`**
   - 修改第 316 行，使用動態的 `backupDir` 變數
   - 如果變數不存在，使用預設值作為後備

#### 修改細節

**BackupRestoreService.js（新增）**：
```javascript
/**
 * 獲取備份目錄路徑（靜態方法，供路由使用）
 */
static getBackupDir() {
  return BACKUP_DIR;
}

/**
 * 獲取安裝目錄路徑（靜態方法，供路由使用）
 */
static getInstallDir() {
  return INSTALL_DIR;
}
```

**backupRestore.js（修改）**：
```javascript
// 獲取備份目錄路徑
let backupDir = '/opt/invoice-bonus-backups'; // 預設值
try {
  backupDir = BackupRestoreService.getBackupDir() || backupDir;
} catch (err) {
  console.warn('[備份還原] 無法獲取備份目錄路徑:', err.message);
}

res.render('backup-restore/index', {
  // ...
  backupDir: backupDir,
  // ...
});
```

**index.ejs（修改）**：
```html
<!-- 修改前 -->
<li>備份檔案存放在 <code>/opt/invoice-bonus-backups</code></li>

<!-- 修改後 -->
<li>備份檔案存放在 <code><%= typeof backupDir !== 'undefined' && backupDir ? backupDir : '/opt/invoice-bonus-backups' %></code></li>
```

## 問題 2：角色編輯更新失敗

### 問題描述
編輯已存在的角色後，點擊「更新角色」按鈕會一直顯示「處理中...」，無法完成更新操作。

### 根本原因
PUT 方法在某些環境或配置下可能不穩定，導致請求無法正確處理或響應。

### 解決方案

#### 修改檔案（2 個）

1. **`src/routes/roles.js`**
   - 將更新邏輯提取為共用函數 `handleRoleUpdate`
   - 同時支持 PUT 和 POST 方法進行更新
   - 添加更詳細的調試日誌

2. **`src/views/roles/form.ejs`**
   - 修改前端提交邏輯，統一使用 POST 方法
   - POST 方法更加穩定可靠

#### 修改細節

**roles.js（修改）**：
```javascript
// 更新角色的共用處理函數
async function handleRoleUpdate(req, res) {
  try {
    console.log('[更新角色] 收到請求，方法:', req.method, ' ID:', req.params.id);
    console.log('[更新角色] 請求 body:', req.body);
    console.log('[更新角色] Content-Type:', req.headers['content-type']);
    
    // ... 原有的更新邏輯 ...
  } catch (err) {
    // ... 錯誤處理 ...
  }
}

// 更新角色 - PUT 方法
router.put('/:id', handleRoleUpdate);

// 更新角色 - POST 方法（作為備選）
router.post('/:id', handleRoleUpdate);
```

**form.ejs（修改）**：
```javascript
// 修改前
const method = ${isEdit ? '"PUT"' : '"POST"'};

// 修改後
const method = 'POST'; // 統一使用 POST 方法，後端會處理新增和更新
```

### 為什麼使用 POST 而不是 PUT？

1. **兼容性更好**：POST 方法被所有瀏覽器和服務器廣泛支持
2. **更穩定**：不會因為中間件配置問題導致請求失敗
3. **調試更容易**：瀏覽器開發者工具對 POST 的支持更完善
4. **符合實踐**：許多現代 Web 應用使用 POST 進行所有修改操作

## 技術細節

### 備份目錄偵測流程

```
1. BackupRestoreService 模組載入
   ↓
2. 調用 detectInstallDir()
   - 檢查當前目錄是否在 /opt 下
   - 嘗試已知的安裝目錄列表
   - 返回 INSTALL_DIR 或 null
   ↓
3. 調用 detectBackupDir(INSTALL_DIR)
   - 根據安裝目錄生成備份目錄名稱
   - 檢查備份目錄是否存在
   - 嘗試其他可能的備份目錄
   - 返回實際存在的目錄或預設路徑
   ↓
4. 設定全域常數
   - INSTALL_DIR = 偵測結果
   - BACKUP_DIR = 偵測結果
   ↓
5. 提供靜態方法
   - getBackupDir() 返回 BACKUP_DIR
   - getInstallDir() 返回 INSTALL_DIR
   ↓
6. 路由調用並傳遞給視圖
   - backupRestore.js 調用 getBackupDir()
   - 將結果傳遞給 EJS 模板
   ↓
7. 視圖動態顯示
   - EJS 模板使用 <%= backupDir %> 顯示實際路徑
```

### HTTP 方法處理

**Express 路由配置**：
```javascript
// 同一個處理函數，支持兩種方法
router.put('/:id', handleRoleUpdate);   // PUT /roles/:id
router.post('/:id', handleRoleUpdate);  // POST /roles/:id
```

**請求路徑**：
- 新增角色：`POST /roles`
- 更新角色：`POST /roles/:id` （原本是 PUT，現在統一用 POST）

**這不會衝突嗎？**
不會。Express 按照以下順序匹配路由：
1. 精確匹配（無參數）：`POST /roles` → 創建
2. 參數匹配（有參數）：`POST /roles/:id` → 更新

## 常見問題

### Q1: 為什麼備份路徑還是顯示舊的？

**A**: 可能的原因：
1. **服務未重啟**：`sudo systemctl restart project-system.service`
2. **瀏覽器快取**：強制重新整理（Ctrl+F5 或 Cmd+Shift+R）
3. **檔案未上傳成功**：檢查檔案修改時間和內容

**檢查方法**：
```bash
# 查看視圖檔案是否包含動態路徑
grep "backupDir" /opt/project-system/src/views/backup-restore/index.ejs

# 應該看到：
# <%= typeof backupDir !== 'undefined' && backupDir ? backupDir : '/opt/invoice-bonus-backups' %>
```

### Q2: 角色更新還是失敗，怎麼辦？

**A**: 檢查步驟：

1. **清除瀏覽器快取**：強制重新整理頁面

2. **查看 Console 日誌**：
   - 按 F12
   - 切換到 Console 面板
   - 提交表單
   - 查看是否有錯誤訊息

3. **查看 Network 請求**：
   - 在開發者工具切換到 Network 面板
   - 提交表單
   - 查找 POST /roles/:id 請求
   - 檢查：
     - Status 是否為 200
     - Response 是否為 `{success: true, ...}`
     - 如果失敗，查看 Response 中的錯誤訊息

4. **查看服務器日誌**：
   ```bash
   sudo journalctl -u project-system.service -f
   ```
   然後執行更新操作，查看即時日誌輸出

5. **確認檔案已更新**：
   ```bash
   # 檢查路由檔案
   grep "handleRoleUpdate" /opt/project-system/src/routes/roles.js
   
   # 檢查視圖檔案
   grep "const method = 'POST'" /opt/project-system/src/views/roles/form.ejs
   ```

### Q3: 為什麼新增角色正常，但編輯就失敗？

**A**: 這通常是因為：
1. 前端沒有正確發送 PUT/POST 請求
2. 後端路由配置問題
3. 資料庫鎖定

**解決方法**：
- 確保已部署最新的 `roles.js` 和 `form.ejs`
- 重啟服務清除任何潛在的資料庫鎖
- 查看詳細日誌定位問題

### Q4: 如何確認備份目錄偵測正確？

**A**: 查看服務啟動日誌：

```bash
sudo systemctl restart project-system.service
sudo journalctl -u project-system.service -n 100 --no-pager | grep -E "INSTALL_DIR|BACKUP_DIR|BackupRestoreService"
```

應該看到偵測過程的日誌，包括找到的安裝目錄和備份目錄。
