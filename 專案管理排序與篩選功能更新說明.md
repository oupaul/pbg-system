# 專案管理排序與篩選功能更新說明

## 更新日期
2026-01-13

## 更新內容

### 1. 未收款篩選功能修復

#### 問題描述
原本的未收款篩選功能只檢查「有開發票但沒有收款記錄」的專案（`total_received = 0`），無法正確篩選出「有部分收款但仍有未收款金額」的專案。

#### 修復內容
- **修正篩選邏輯**：從 `(total_invoiced > 0) AND (total_received = 0 OR total_received IS NULL)` 
  改為 `(total_invoiced > 0) AND (total_invoiced - COALESCE(total_received, 0) - COALESCE(sales_discount, 0) > 0)`
- **正確計算未收款金額**：考慮已開立發票、已收款金額和銷貨折讓
- **支援部分收款情況**：現在能正確篩選出所有有未收款金額的專案

#### 影響範圍
- 專案管理頁面的「未收款」快速篩選按鈕
- 專案管理頁面的未收款篩選功能

#### 修改檔案
- `src/models/Project.js`：修正 `findAll` 方法中的未收款篩選邏輯

---

### 2. 排序功能增強

#### 新增功能
為專案管理頁面新增三個欄位的排序功能：

1. **未開發票** (`uninvoiced_amount`)
   - 支援升序/降序排序
   - 點擊表頭可切換排序方向

2. **未收款** (`unpaid_amount`)
   - 支援升序/降序排序
   - 使用計算表達式：`(total_invoiced - COALESCE(total_received, 0) - COALESCE(sales_discount, 0))`
   - 點擊表頭可切換排序方向

3. **預計開票** (`expected_invoice_year_month`)
   - 支援升序/降序排序
   - 點擊表頭可切換排序方向

#### 技術實現
- 在 `Project.js` 的 `validSortFields` 中新增三個排序欄位
- 對於 `unpaid_amount`，由於視圖中沒有此欄位，在 ORDER BY 中使用計算表達式
- 在路由中新增排序連結和圖示生成
- 在視圖中將表頭改為可點擊的排序連結

#### 影響範圍
- 專案管理頁面表頭
- 專案列表排序功能

#### 修改檔案
- `src/models/Project.js`：新增排序欄位支援
- `src/routes/projects.js`：新增排序連結和圖示
- `src/views/projects/index.ejs`：更新表頭為可排序連結

### 3. 修復腳本動態化改進

#### 問題描述
`fix-roles-table.sh` 和 `fix-user-role-constraint.sh` 兩個腳本中硬編碼了專案目錄和服務名稱，無法適應不同的部署環境。

#### 修復內容

**fix-roles-table.sh**
- 保留動態選擇目錄功能（`list_install_dirs`）
- 新增：從 `deploy.config.sh` 讀取服務名稱
- 如果配置文件不存在或沒有服務名稱，從 systemd 服務文件推斷（檢查 `WorkingDirectory`）
- 如果還是找不到，使用目錄名稱作為備選

**fix-user-role-constraint.sh**
- 移除硬編碼目錄列表（`/opt/project-system`、`/opt/invoice-bonus-system` 等）
- 改用動態檢測目錄（使用與 `fix-roles-table.sh` 相同的 `list_install_dirs` 函數）
- 載入 `deploy.config.sh` 配置文件（如存在）
- 移除硬編碼服務名稱列表
- 改為從配置文件或 systemd 服務文件推斷服務名稱
- 改進服務停止和啟動的檢查邏輯，確保服務存在後再操作

#### 改進效果
- 腳本能自動適應不同的部署環境
- 支援自訂安裝目錄和服務名稱
- 提升腳本的可移植性和兼容性

#### 修改檔案
- `fix-roles-table.sh`：動態讀取配置和服務名稱
- `fix-user-role-constraint.sh`：動態檢測目錄和服務名稱

## 使用說明

### 未收款篩選
1. 進入專案管理頁面
2. 點擊「未收款」快速篩選按鈕
3. 系統會顯示所有有未收款金額的專案（包括部分收款的情況）

### 排序功能
1. 進入專案管理頁面
2. 點擊「未開發票」、「未收款」或「預計開票」表頭
3. 第一次點擊為升序，再次點擊為降序
4. 表頭會顯示排序方向箭頭（↑ 升序，↓ 降序）

### 修復腳本使用
兩個修復腳本現在會自動：
1. 掃描 `/opt` 目錄下的所有專案安裝目錄
2. 讓用戶選擇要操作的目錄
3. 自動讀取該目錄的 `deploy.config.sh` 配置文件
4. 從配置文件中獲取服務名稱，或從 systemd 服務文件推斷
5. 執行相應的修復操作

## 測試建議

### 未收款篩選測試
1. 建立一個有部分收款的專案（已開發票，但只收部分款項）
2. 點擊「未收款」篩選按鈕
3. 確認該專案出現在篩選結果中

### 排序功能測試
1. 點擊「未開發票」表頭，確認排序正常
2. 點擊「未收款」表頭，確認排序正常
3. 點擊「預計開票」表頭，確認排序正常
4. 確認排序方向箭頭正確顯示

### 腳本測試
1. 在不同部署環境下執行 `fix-roles-table.sh`
2. 確認腳本能正確找到專案目錄和服務名稱
3. 執行 `fix-user-role-constraint.sh`
4. 確認腳本能正確找到專案目錄和服務名稱

## 技術細節

### 未收款金額計算公式
```
未收款金額 = 已開立發票金額 - 已收款金額 - 銷貨折讓
```

### 排序欄位處理
- `uninvoiced_amount`：直接使用視圖中的欄位
- `unpaid_amount`：使用計算表達式 `(total_invoiced - COALESCE(total_received, 0) - COALESCE(sales_discount, 0))`
- `expected_invoice_year_month`：直接使用視圖中的欄位

### 服務名稱推斷順序
1. 從 `deploy.config.sh` 配置文件讀取 `SERVICE_NAME`
2. 從 systemd 服務文件推斷（檢查 `WorkingDirectory` 是否匹配專案目錄）
3. 使用目錄名稱作為備選（`basename "$PROJECT_DIR"`）
