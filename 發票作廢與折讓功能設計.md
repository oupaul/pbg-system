# 發票作廢與折讓功能設計

## 需求概述

- **發票作廢重開**：原發票作廢後另開新發票，保留原發票與新發票的關聯
- **整筆折讓**：發票金額整筆折讓（銷貨折讓／折讓單），需保留原發票資訊
- **保留軌跡**：上述變更均需保留完整歷史紀錄，可供查詢與稽核

---

## 設計方案

### 一、資料庫結構調整

#### 1. invoices 表新增欄位

| 欄位 | 類型 | 說明 |
|------|------|------|
| `status` | TEXT | 發票狀態：`有效`、`作廢`、`整筆折讓`（預設 `有效`） |
| `voided_at` | TEXT | 作廢／折讓日期（datetime） |
| `void_reason` | TEXT | 作廢或折讓原因（可選） |
| `replacement_invoice_id` | INTEGER | 作廢重開時：新發票 ID（作廢的發票指向新發票） |
| `original_invoice_id` | INTEGER | 作廢重開時：原發票 ID（新發票指向被作廢的發票） |

**關聯關係：**
```
作廢重開流程：
原發票 A (status=作廢) ──replacement_invoice_id──> 新發票 B (status=有效)
新發票 B ──original_invoice_id──> 原發票 A

整筆折讓：
發票 C (status=整筆折讓) — 無 replacement，保留原金額供查詢
```

#### 2. 遷移腳本

- 檔案：`migrations/migrate_invoice_status.js`
- 為既有發票補上 `status = '有效'`
- 新增欄位皆可為 NULL，既有資料不受影響

---

### 二、計算邏輯調整

#### 1. 已開發票總額（total_invoiced）

僅計算 `status = '有效'` 的發票：

```sql
-- 修正前
SELECT COALESCE(SUM(amount_with_tax), 0) FROM invoices WHERE project_id = ?

-- 修正後
SELECT COALESCE(SUM(amount_with_tax), 0) FROM invoices 
WHERE project_id = ? AND (status IS NULL OR status = '有效')
```

#### 2. 影響範圍

- `Invoice.getTotalByProject()` 
- `v_project_summary` 視圖
- 儀表板、專案列表、獎金計算相關查詢

---

### 三、業務流程

#### 1. 發票作廢（不重開）

- 操作：將發票狀態改為 `作廢`
- 填寫：`voided_at`、`void_reason`（選填）
- 軌跡：`AuditLogService.logUpdate` 記錄
- 效果：該發票不再計入 `total_invoiced`

#### 2. 發票作廢並重開

- 步驟 1：將原發票狀態改為 `作廢`，填寫 `voided_at`、`void_reason`
- 步驟 2：建立新發票（可帶入原發票部分資料）
- 步驟 3：設定關聯：
  - 原發票：`replacement_invoice_id` = 新發票 ID
  - 新發票：`original_invoice_id` = 原發票 ID
- 軌跡：`create` + `update` 皆由 AuditLog 記錄

#### 3. 整筆折讓

- 操作：將發票狀態改為 `整筆折讓`
- 填寫：`voided_at`、`void_reason`（選填）
- 效果：保留原發票與金額，但不計入 `total_invoiced`
- 軌跡：`AuditLogService.logUpdate` 記錄

---

### 三之一、整筆折讓是否造成重複？說明

**結論：不會造成已開立發票金額或應收款項重複。**

| 疑慮 | 說明 |
|------|------|
| 已開立發票金額重複？ | **不會。** 整筆折讓是「同一筆」發票僅改狀態，不新增第二筆發票。`total_invoiced` 只加總 `status = '有效'`，該筆被排除，所以這筆金額根本不會被算進已開立總額，不會被算兩次。 |
| 應收款項重複？ | **不會。** 應收 = 已開立 − 已收款 − 銷貨折讓。已開立已排除整筆折讓，該筆不會進已開立，自然也不會進應收，無重複。 |

**設計要點：**

1. **單一紀錄、排除加總**  
   整筆折讓 = 原發票一筆紀錄改為 `status = '整筆折讓'`，金額保留在該筆僅供查詢／稽核。所有「已開立發票總額」的計算都必須加上條件：只計 `status = '有效'`（或 `status IS NULL` 視同有效）。這樣該筆就不會進入加總，不會重複。

2. **與「銷貨折讓」的區分，避免雙重扣除**  
   - **整筆折讓（發票層級）**：該發票不計入 `total_invoiced`，等於從「已開立」直接拿掉這筆，不另外再扣一筆。  
   - **銷貨折讓（專案層級 `projects.sales_discount`）**：專案上手動輸入的折讓金額，在「應收 = 已開立 − 已收款 − sales_discount」時再扣。  
   - **同一筆錢只應擇一處理**：若某發票已標記為「整筆折讓」，就不要再把同一筆金額填到該專案的 `sales_discount`，否則會等於扣兩次，造成已開立／應收偏低。

3. **實作時必須全面篩選 status**  
   所有計算 `total_invoiced`、未收款、獎金相關的查詢（含 `Invoice.getTotalByProject`、`v_project_summary`、儀表板、專案列表、匯出等）都要加上「只計有效發票」的條件；否則會變成「漏排除」，整筆折讓／作廢發票仍被算進去，導致**已開立與應收偏高**（不是重複，而是多算）。

### 四、UI 設計建議

#### 1. 發票列表顯示

| 狀態 | 顯示方式 | 說明 |
|------|----------|------|
| 有效 | 一般列 | 正常發票 |
| 作廢 | 灰色列 + 標籤「作廢」 | 可顯示「→ 重開 #123」連結 |
| 整筆折讓 | 灰色列 + 標籤「整筆折讓」 | 保留原金額供檢視 |

#### 2. 操作按鈕

- **作廢**：單一發票 → 彈窗填寫原因 → 確認後更新狀態
- **作廢重開**：單一發票 → 彈窗填寫原因 → 開啟新發票表單（預填部分欄位）→ 送出後建立新發票並寫入關聯
- **整筆折讓**：單一發票 → 彈窗填寫原因 → 確認後更新狀態

#### 3. 軌跡查詢

- 修改記錄：沿用現有 `audit_logs` 顯示 create / update / delete
- 作廢／折讓紀錄：可由 `status`、`voided_at`、`void_reason` 查詢
- 作廢重開關聯：顯示 `original_invoice_id` ↔ `replacement_invoice_id` 鏈結

---

### 五、實作順序建議

1. **Phase 1：資料庫**
   - 新增遷移腳本
   - 更新 v_project_summary 等視圖

2. **Phase 2：Model / 查詢**
   - 更新 `Invoice.js`：create / update / findByProject / getTotalByProject
   - 新增 `void()`、`voidAndReissue()`、`setAllowance()` 等方法

3. **Phase 3：API / 路由**
   - `POST /invoices/:id/void`（作廢）
   - `POST /invoices/:id/void-and-reissue`（作廢重開）
   - `POST /invoices/:id/allowance`（整筆折讓）

4. **Phase 4：UI**
   - 發票列表狀態標籤與樣式
   - 作廢／作廢重開／整筆折讓按鈕與表單
   - 關聯發票的連結顯示

5. **Phase 5：報表與匯出**
   - 依需求決定是否在匯出時區分或篩選「有效」發票
   - 獎金計算僅納入有效發票（沿用 Phase 1 計算邏輯）

---

### 六、軌跡保留方式

| 項目 | 方式 |
|------|------|
| 操作紀錄 | 既有 `system_logs`（create / update） |
| 狀態變更 | `status`、`voided_at`、`void_reason` 欄位 |
| 作廢重開關聯 | `original_invoice_id`、`replacement_invoice_id` |
| 查詢作廢／折讓紀錄 | `WHERE status IN ('作廢','整筆折讓')` |

既有 `AuditLogService` 會記錄欄位變更（含 status），無需新增表即可滿足軌跡需求。

---
