# 自動備份排程檢查與修正指南

## 問題描述

一鍵部署（deploy.sh）完成後，自動排程備份可能未正確設定或未顯示下次執行時間。

## 一、檢查方式

### 1. 確認備份 Timer 是否存在

```bash
# 根據您的服務名稱，替換 project-system-prod 為實際服務名稱
sudo systemctl list-unit-files | grep backup

# 應顯示類似：
# project-system-prod-backup.service  static
# project-system-prod-backup.timer    enabled
```

### 2. 檢查 Timer 是否啟用

```bash
# 替換為您的備份 timer 名稱（通常為 服務名稱-backup.timer）
sudo systemctl is-enabled project-system-prod-backup.timer

# 應顯示：enabled
```

### 3. 檢查下次執行時間

```bash
sudo systemctl list-timers | grep backup

# 正常情況應顯示 NEXT 與 LEFT 欄位有時間，例如：
# Thu 2026-02-05 02:02:57 CST  17h  project-system-prod-backup.timer
```

若 **NEXT** 與 **LEFT** 顯示為 `-`，表示 Timer 雖已建立但 systemd 尚未正確計算下次觸發時間。

### 4. 檢查 Timer 詳細狀態

```bash
sudo systemctl status project-system-prod-backup.timer

# 若顯示 "Trigger: n/a"，表示需重新載入
```

### 5. 確認備份腳本權限

```bash
# 前往安裝目錄（依實際路徑調整）
ls -la /opt/專案目錄名稱/backup.sh
ls -la /opt/專案目錄名稱/setup-backup-timer.sh
ls -la /opt/專案目錄名稱/deploy.config.sh

# 應具有執行權限（-rwxr-xr-x）
```

---

## 二、修正方式

### 方法 A：重新載入並啟動 Timer（最常見）

若 Timer 已存在但 NEXT 顯示 `-`：

```bash
# 1. 重新載入 systemd 配置
sudo systemctl daemon-reload

# 2. 重新啟動備份 timer（替換為您的 timer 名稱）
sudo systemctl restart project-system-prod-backup.timer

# 3. 確認下次執行時間
sudo systemctl list-timers | grep backup
```

### 方法 B：修正「無效的選擇」重複錯誤（排程備份卡在互動選單）

若 `journalctl -u 服務名-backup.service` 顯示大量「無效的選擇，請輸入 1-2 之間的數字」，表示 systemd 執行 backup.sh 時未以非互動模式運行。

**解法 1：重新執行 setup-backup-timer.sh**（推薦，會產生正確的 ExecStart）

```bash
cd /opt/專案目錄名稱   # 例如 /opt/project-system-prod
sudo ./setup-backup-timer.sh 1   # 或選擇其他備份頻率

# 更新後重新載入
sudo systemctl daemon-reload
sudo systemctl restart 服務名-backup.timer
```

**解法 2：手動編輯 service 檔**

```bash
sudo nano /etc/systemd/system/服務名-backup.service
```

將 `ExecStart=` 改為（路徑依實際安裝目錄調整）：

```ini
ExecStart=/bin/bash -c 'NON_INTERACTIVE=1 exec /opt/專案目錄/backup.sh'
```

儲存後執行 `sudo systemctl daemon-reload`，再測試備份。

### 方法 C：手動設定自動備份

若部署時未設定或設定失敗：

```bash
# 前往安裝目錄
cd /opt/專案目錄名稱   # 例如 /opt/project-system-prod

# 執行備份設定腳本（可自訂時間）
sudo ./setup-backup-timer.sh

# 依提示選擇：
# 1) 每日凌晨 2:00
# 2) 每週日凌晨 2:00
# 3) 每日 2:00 和 14:00
# 4) 自訂 OnCalendar 格式
# 6) 每日自訂時間（輸入 時:分，如 03:30）
```

### 方法 D：確保腳本權限後重新設定

```bash
cd /opt/專案目錄名稱

# 設定執行權限
sudo chmod 755 deploy.config.sh
sudo chmod +x backup.sh
sudo chmod +x setup-backup-timer.sh

# 重新執行備份設定
sudo ./setup-backup-timer.sh
```

### 方法 E：立即測試備份是否正常

```bash
# 手動觸發一次備份
sudo systemctl start project-system-prod-backup.service

# 查看執行結果
sudo journalctl -u project-system-prod-backup.service -n 50 --no-pager
```

---

## 三、自訂備份時間

執行 `sudo ./setup-backup-timer.sh` 後可選擇：

| 選項 | 說明 |
|------|------|
| 1 | 每日凌晨 2:00 |
| 2 | 每週日凌晨 2:00 |
| 3 | 每日 2:00 和 14:00 |
| 4 | 進階自訂（OnCalendar 格式） |
| 5 | 停用自動備份 |
| 6 | **每日自訂時間**（輸入 時:分，如 `03:30` 表示每天 3:30） |

**範例：每天凌晨 3:30 備份**
- 選擇選項 6
- 輸入 `03:30` 或 `3:30`

---

## 四、常見原因整理

| 現象 | 可能原因 | 解決方式 |
|------|----------|----------|
| 日誌出現「無效的選擇，請輸入 1-2 之間的數字」重複 | 排程備份時 backup.sh 進入互動模式（缺少 NON_INTERACTIVE） | 重新執行 setup-backup-timer.sh 以更新 service 檔案 |
| list-timers 顯示 NEXT 為 `-` | systemd 未正確計算觸發時間 | 執行 daemon-reload + restart |
| 找不到 backup.timer | 部署時未執行或跳過設定 | 手動執行 setup-backup-timer.sh |
| backup.sh 執行失敗 | 腳本無執行權限 | chmod +x backup.sh |
| deploy.config.sh 讀取失敗 | 權限或路徑錯誤 | chmod 755 deploy.config.sh |

---

## 五、部署後建議檢查清單

1. `sudo systemctl list-timers | grep backup` 確認 NEXT 有顯示時間
2. `sudo systemctl start 服務名-backup.service` 手動執行一次測試
3. `journalctl -u 服務名-backup.service -n 20` 確認無錯誤訊息
